## Что это за проект
DBT-проект, в котором логической моделью данных (далее ЛМД) является Data Vault (далее DV). В качестве хранилища используется БД Postgres.

## Компоненты конкретной отдельной модели
Каждая DBT-модель, которая является частью ЛМД DV должна состоять из 3-х обязательных файлов
- SQL-файл: `/models/scripts/cdm/<model_name>.sql`
- YML-файл: `/models/schemas/cdm/<model_name>.yml`
- MD-файл: `/docs/cdm/<model_name>.md`

Данное правило применяется только к DV-слою (`cdm`). Остальные слои (`ods`, `dm`, `service`) живут по другим правилам:
- В слое `ods` лежат источники данных, на основе которых строятся DBT-модели
- В слое `dm` лежат денормализованные витрины данных для аналитики, которые строятся на основе моделей DV-слоя (лежат в CDM)
- В слое `service` лежит сервисная модель, которая необходима для построения Semantic Layer

Для создания конфигурации модели (YAML-файл, MD-файл) ДОЛЖЕН использоваться скилл `make-model-config`. Для создания ТОЛЬКО документации модели ДОЛЖЕН использоваться скилл `create-doc`

## Соглашения внутри проекта
1. Каждая модель имеет свой префикс в нейминге:
    - Хабы: `h_*.sql, h_*.yml, h_*.md`
    - Линки: `l_*.sql, l_*.yml, l_*.md`
    - Сателлиты: `s_*.sql, s_*.yml, s_*.md`
    - Бриджи: `b_*.sql, b_*.yml, b_*.md`

2. Хэширование реализовано через `MD5`
Все ключи, которые требуют хэширования, хэшируются с помощью `MD5`. Перед хэшированием поле ключа приводится к верхнему регистру через `UPPER` и очищается от пробелов c `TRIM`.

    В зависимости от типа таблицы, используются следующие способы получения хэш-ключа:
    - Хабы: 
        - Хэширование первичного ключа бизнес-сущности с обработкой - `MD5(UPPER(TRIM(<primary_key_name>)))`. Примеры - `h_user.sql`, `h_order.sql`
    - Линки: 
        - Хэширование конкатенированных вторичных ключей объединяемых бизнес-сущностей (link_id) - `MD5(UPPER(TRIM(<foreign_key_name_1>)) || '||' || UPPER(TRIM(<foreign_key_name_2>)))`. Пример - `l_user_x_order.sql`
        - Хэширование вторичных ключей бизнес-сущностей - такое же, как в пункте "Хабы". Пример - `l_user_x_order.sql`
    - Сателлиты: 
        - Хэширование вторичного ключа бизнес-сущности - такое же, как в пункте "Хабы". Пример - `s_user.sql`
        - Хэширование поля mt_hash_diff - `MD5(
            COALESCE(<attr_1>, '') || '||' ||
            COALESCE(<attr_2>, '') || '||' ||
            COALESCE(<attr_3>, '') || '||' ||
            COALESCE(<attr_4>, '')
        )`. Пример - `s_user.sql`
    - Бриджи:
        - Хэширование конкатенированных вторичных ключей объединяемых бизнес-сущностей (bridge_link_id) - `MD5(
            COALESCE(<foreign_key_name_1>, '') || '||' ||
            COALESCE(<foreign_key_name_2>, '') || '||' ||
            COALESCE(<foreign_key_name_3>, '') || '||' ||
            COALESCE(<foreign_key_name_4>, '')
        )`. Пример - `b_order_x_user_x_retailer_x_courier.sql`

3. Поля в SQL-моделях имеют определенный унифицированный тип данных:
    - Текстовые - varchar
    - Целочисленные - integer
    - Дата и время - timestamptz
    - Дробные - Numeric

    В полях в SQL-файлах ДОЛЖЕН явно задаваться каст для каждого из полей. Пример для разных полей

    ```sql
    SELECT 
        CAST(<text_field_name> AS VARCHAR) AS text_field_name,
        CAST(<int_field_name> AS INTEGER) AS int_field_name,
        CAST(<timestamp_field_name> AS TIMESTAMPTZ) AS timestamp_field_name
    ```
4. Все названия мета-полей имеют префикс `mt_*`. У каждого типа таблицы в DV есть обязательный набор мета-полей:

    - Хабы и Линки:`mt_src_id`, `mt_load_dttm`
    - Сателлиты: такие же, как у Хабов и Линков + `mt_hash_diff`, `mt_valid_from_dttm`, `mt_valid_to_dttm`, `mt_active_flg`, `mt_deleted_flg`
    - Бриджи: такие же, как у Хабов и Линков

5. Обязательные тесты по типу сущности                          
    - `h_*`: unique + not_null на хэш-ключ.                                                                                                   
    - `l_*`, `b_*`: unique на ключ + relationships к каждому связанному h_*.                                               
    - `s_*`: duplicates_active_rows_check + проверка mt_valid_from_dttm <= mt_valid_to_dttm.

6. В сателлитах всегда используется SCD-материализация. Лежит в `/macros/materializations/scd_materialization.sql`

- Для создания документации к модели: `.claude/skills/create-doc/SKILL.md`
