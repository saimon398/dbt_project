## Алгоритм действий

1. Найти и прочитать SQL-файл модели `$model_name` в `models/`. Если файл не найден — прервать выполнение и сообщить пользователю.
2. Проанализировать SQL: определить колонки, типы данных, источники (`ref()` / `source()`), бизнес-логику и тип модели (hub / link / satellite / mart / staging и т.п.).
3. Уточнить у пользователя назначение / бизнес-смысл модели, если это не очевидно из SQL и комментариев.
4. Уточнить материализацию, если она не задана в `config()` модели или в `dbt_project.yml` (по умолчанию — `table`).
5. Проверить, существует ли уже MD-файл для модели. Если да — спросить пользователя, перезаписать его или дополнить.
6. Создать MD-файл для `$model_name` по правилам ниже.
7. Напомнить пользователю, что для отображения описания в dbt docs нужно сослаться на блок `{% docs %}` из соответствующего `.yml`-файла через `description: '{{ doc("model_name") }}'` (либо предложить вызвать скилл `make-model-config`).

## Правила размещения файлов

MD-файлы располагаются в `docs/`. Внутри `docs/` структура папок повторяет расположение SQL-модели относительно `models/scripts`.

Пример: модель `h_user.sql` лежит в `models/scripts/cdm/` → MD-файл создаётся в `docs/cdm/h_user.md`.

Имя MD-файла должно совпадать с именем SQL-модели.

## Обязательные секции описания

Любое описание модели должно содержать:

- **Назначение** — что за сущность/витрина, какой бизнес-смысл.
- **Источник** — таблицы и модели, на которых построена (определяются по `ref()` / `source()` в SQL).
- **Схема** — в какой схеме материализуется модель.
- **Материализация** — `table` / `view` / `incremental` / кастомная (`scd` и т.п.) с пояснением логики загрузки, если она нетривиальна.
- **Особенности** — ключи, инкрементальная стратегия, SCD-логика, фильтры — то, что важно знать потребителю.

### Hub

```
#### Описание модели
Таблица-хаб "Пользователь". Хранит суррогатный ключ пользователя, вычисленный из поля `user_src_id`.

Источником для модели служит таблица `paas_content_demand__mindbox.spree_users`.

Сама модель располагается в схеме `cdm`.

Материализация — инкрементальная. Если мануально не запускается полная перезапись, в таблицу подгружаются только новые бизнес-ключи сущности "Пользователь".

{% enddocs %}
```

### Link
```
{% docs l_user_x_order %}

#### Описание модели
Таблица-линк, связывающая хабы "Пользователь" и "Заказ". Хранит суррогатные ключи обеих сущностей и хеш связи.

Источниками служат: `ref('h_user')`, `ref('h_order')` и исходная таблица заказов.

Схема — `cdm`. Материализация — инкрементальная: добавляются только новые комбинации ключей.

{% enddocs %}
```

### Satellite
```
{% docs s_user %}

#### Описание модели
Саттелит хаба "Пользователь". Хранит описательные атрибуты пользователя и историю их изменений.

Источник — `ref('h_user')` и `paas_content_demand__mindbox.spree_users`.

Схема — `cdm`. Материализация — `scd` (кастомная): новая запись добавляется только при изменении атрибутов.

{% enddocs %}
```
