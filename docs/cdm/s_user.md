#### Описание модели
Сателлит хаба "Пользователь". Хранит описательные атрибуты пользователя (имя, фамилия, email, телефон) и историю их изменений во времени. Используется потребителями для получения актуального либо исторического состояния пользователя на заданную дату.

#### Источник
- `ref('h_user')` — связан логически: сателлит расширяет хаб (зависимость задана в `depends_on`).
- `source('paas_content_demand__mindbox', 'spree_users')` — источник атрибутов и поля `valid_from`.

#### Схема
Модель материализуется в схеме `cdm`.

#### Материализация
Кастомная — `scd_materialization` (см. `macros/materializations/scd_materialization.sql`). Логика загрузки:
- При первом запуске или `--full-refresh` целевая таблица создаётся через CTAS из временной таблицы.
- При инкрементальном запуске:
  - вставляются записи с новыми бизнес-ключами;
  - вставляются записи, у которых для существующего ключа изменился `mt_hash_diff`;
  - у предыдущей активной версии проставляется `mt_valid_to_dttm = новый mt_valid_from_dttm` и `mt_active_flg = false`;
  - записи, которых уже нет в источнике, помечаются как удалённые (`mt_deleted_flg = true`, `mt_active_flg = false`, `mt_valid_to_dttm = current_timestamp`).

Параметр материализации: `hub_hash_key: user_id`.

#### Особенности
- `mt_hash_diff` — `md5` от конкатенации `email`, `first_name`, `last_name`, `phone`. Изменение любого из этих полей запускает создание новой версии записи.
- В отличие от других сателлитов, `mt_valid_from_dttm` берётся напрямую из источника (`valid_from`), а не из момента загрузки. Это позволяет фиксировать реальное время изменения данных в исходной системе.
- `mt_active_flg` — флаг текущей активной версии для пары `(user_id, mt_hash_diff)`.
- Тесты: проверка уникальности пары `(user_id, mt_valid_from_dttm)`, валидность диапазона `mt_valid_from_dttm <= mt_valid_to_dttm`, `not_null` и проверка отсутствия дубликатов активных строк, `relationships` на `h_user.user_id`.
