#### Описание модели
Единая позаказная витрина для BI с атрибутами заказа, ритейлера, пользователя и курьера на момент анализа. Используется как источник для отчётности по доставкам.

#### Источник
Осью витрины служит бридж `ref('b_order_x_user_x_retailer_x_courier')`, к которому подтягиваются только активные версии атрибутов из сателлитов:
- `ref('s_order')` — атрибуты заказа (`order_price`, `order_created_dttm`, `order_shipped_dttm`, `shipment_type`);
- `ref('s_retailer')` — атрибуты ритейлера (`retailer_name`, `retailer_type`);
- `ref('s_user')` — атрибуты пользователя (`user_first_name`, `user_last_name`, `user_email`);
- `ref('s_courier')` — атрибуты курьера (`courier_type`).

Все сателлиты фильтруются по `mt_active_flg = true`, поэтому витрина отражает текущее состояние, а не историю.

#### Схема
`dm`.

#### Материализация
Кастомная `batch_materialization` с шардированием по `order_id` на 8 батчей (`column_name: order_id`, `batch_count: 8`). При запуске таблица пересобирается полностью, но загрузка идёт партициями — это снижает пиковую нагрузку на Postgres и ограничивает объём данных, удерживаемый в одной транзакции.

#### Особенности
- `order_id` — первичный ключ витрины, на нём стоит `unique`-тест.
- `retailer_id`, `user_id`, `courier_id` — `not_null`, так как берутся из бриджа и обязаны существовать.
- На уровне модели работает тест `dbt_expectations.expect_table_row_count_to_equal_other_table` против `b_order_x_user_x_retailer_x_courier`: количество строк витрины должно совпадать с количеством связей в бридже — это гарантирует, что `LEFT JOIN`-ы по сателлитам не теряют и не дублируют записи.
- Контракт `contract.enforced: true` зафиксирован: типы колонок (`varchar`, `numeric(10, 2)`, `timestamptz`) изменить без ошибки нельзя.
